# Deployment

Production runs on cPanel hosting (`animecream.com`), not Docker/Kubernetes/a cloud provider — this file describes that real mechanism only.

## Two independent things get deployed

1. **`module-api`'s own application code** (this repo).
2. **The database schema** (`animecream-data`, `auth-data`, `finan-data` — three separate sibling repos, see `docs/specification-roadmap.md`'s multi-repo note).

They ship through similar-looking but separate pipelines, both under a `com/` folder of batch scripts + a gitignored `keys/` folder of credential files, synced to production via a Google Drive Desktop–mounted FTP folder.

## Deploying `module-api`'s code

```bash
npm run bup   # = npm run build && call com/db-sync-FTP-both.bat
```

Builds TypeScript to `dist/`, then FTP-syncs the built output up via `com/db-sync-FTP-both.bat` (the same sync mechanism the schema repos use, see below). This repo's own `.env` on the production server is edited directly (by SSH/FTP), not templated or regenerated by this script — see `docs/setup.md` for the real variable names. **Not independently verified this session**: whether the production Node process picks up new code automatically or needs a manual restart (e.g. via cPanel's "Setup Node.js App" panel) — confirm with whoever manages that panel before relying on this.

## Deploying the database schema

Each of the three sibling schema repos has its own `db-deploy-schema.bat` at its root, running a 4-step, confirmation-gated pipeline (build the incremental SQL → test locally → FTP-sync → apply to the live production database via SSH). **Full mechanics — exact scripts, the non-interactive-invocation quirks, the real success signal to trust, credential file naming — are documented in `docs/specification-roadmap.md` Phase 5b.** Don't duplicate that here; it was written from a real, verified production run and updating two copies would just let them drift.

Short version: `db-structure.sql` is bootstrap-only (destructive, only for a fresh database); `db-views-procs.sql` and `db-trigger.sql` are idempotent and re-run on every deploy; `sql/migrations/000N_*.sql` are incremental and never edited after being committed.

## Image file storage (cPanel symlink)

Series cover images are physically stored in the public web root and mirrored into the upload path via a symlink, so the app can keep saving to its normal `uploads/series/img/tarjeta` path while the files are actually served from a publicly-accessible folder:

- Real folder: `/home/animecre/public_html/webroot/img/tarjeta`
- Symlinked from: `/home/animecre/info.animecream.com/uploads/series/img/tarjeta`

```bash
ln -s /home/animecre/public_html/webroot/img/tarjeta /home/animecre/info.animecream.com/uploads/series/img/tarjeta
```

## What this file deliberately does not cover

Generic cloud/container deployment (AWS/GCP/Azure/Kubernetes/a from-scratch CI/CD pipeline, PM2, nginx load balancing) — none of that is how this project actually runs today. If the app is ever migrated to one of those, rewrite this file against the new real mechanism rather than resurrecting the old generic version of it.
